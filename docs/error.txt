ARCHITECTURE OVERVIEW — TOTAL RECALL ENGINE v5.0

LAYER 1 — PROXY VALIDATION ENGINE (PRE-FLIGHT MANDATORY):
File: core/proxy_engine.py
- Fungsi: validate_proxy(proxy_url) -> ProxyStatus
- Test setiap proxy sebelum masuk pool:
  * Connect test ke httpbin.org/ip atau ipinfo.io/json via proxy
  * Timeout: max 5 detik. Gagal = discard.
  * Anonymity check: bandingkan IP response dengan IP asli attacker. 
    Jika sama = transparent (discard). Jika beda = elite/anonymous (keep).
  * Speed test: measure TTFB (Time To First Byte). Sort proxy pool by speed.
  * Protocol support: HTTP, HTTPS, SOCKS4, SOCKS5. Auto-detect dari URL 
    scheme (http://, https://, socks4://, socks5://).
  * Health check interval: re-check setiap 60 detik selama attack berjalan. 
    Proxy yang mati di-remove dari pool secara real-time.
  * Proxy pool minimum: jika proxy aktif < 10, pause attack dan warning 
    "INSUFFICIENT PROXY POOL".
- Output: proxy pool yang 100% verified, sorted by latency, dengan 
  anonymity level.

LAYER 2 — TARGET INTELLIGENCE & CMS FINGERPRINTING:
File: core/intel_engine.py
- Fungsi: analyze_target(url) -> TargetProfile
- Real CMS Detection (bukan "GENERIC" default):
  * WordPress: cek /wp-includes/wlwmanifest.xml (200 + XML), 
    /wp-json/wp/v2/ (200 + JSON), meta generator tag "WordPress", 
    /wp-content/plugins/ (200 atau 403, bukan 404).
  * Joomla: cek /administrator/ (200/403 dengan Joomla cookie), 
    /media/system/js/ (200), /templates/ (200).
  * Drupal: cek /misc/drupal.js (200), /sites/default/ (403/200), 
    X-Generator header "Drupal".
  * Magento: cek /skin/frontend/ (200), /js/mage/ (200), 
    /media/catalog/ (200).
  * Laravel: cek /vendor/laravel/ (404 tapi pattern), 
    /storage/framework/ (403), X-Powered-By "Laravel".
  * Generic: hanya jika SEMUA check di atas gagal (404 semua).
- Endpoint Ranking:
  * Cek 10 endpoint umum: /, /login, /admin, /wp-admin, /user, /search, 
    /api/, /xmlrpc.php, /wp-json/, /robots.txt.
  * Measure response time per endpoint via HEAD request (bukan GET untuk 
    mengurangi load).
  * Score = (1/response_time_ms) * weight_endpoint. 
    Endpoint admin/login lebih berat weight-nya untuk bypass/WAF testing.
  * Return top 3 endpoint dengan score tertinggi.
- WAF/CDN Detection:
  * Cek header: cf-ray (Cloudflare), x-sucuri-id (Sucuri), 
    server: cloudflare, x-amz-cf-id (CloudFront).
  * Cek response body: "Attention Required", "Cloudflare", "DDoS-GUARD".
  * Return WAF type dan rekomendasi bypass technique.

LAYER 3 — ADAPTIVE ATTACK ENGINE (REAL TRAFFIC, NO FAKE):
File: core/attack_engine.py
- Library WAJIB: curl_cffi (bukan requests, bukan urllib). 
  curl_cffi mendukung TLS impersonation Chrome 120, JA3 fingerprint, HTTP/2.
- Attack Methods:
  * http_get_flood: Kirim GET request massal dengan header real browser 
    (Chrome 120, Firefox 121, Edge 120). Rotate User-Agent per request. 
    Rotate proxy per request. Randomize query parameter untuk bypass cache.
  * http_post_flood: POST dengan body random/JSON/form-data. 
    Content-Type rotation. Bypass CSRF dengan cookie jar dari curl_cffi.
  * slowloris: Kirim HTTP request partial, hold connection terbuka 
    selama mungkin. Send header periodic untuk keep-alive. 
    Target: exhaust connection pool server.
  * bypass_path_flood: Kirim request ke path yang sering tidak di-cache 
    atau tidak di-rate-limit oleh WAF: /?a=1, /wp-json/wp/v2/users, 
    /xmlrpc.php, /?s=random, /feed/.
  * bypass_cf: Jika Cloudflare terdeteksi, gunakan teknik: 
    - real browser session (curl_cffi dengan cookie persistence)
    - request ke origin IP (bypass DNS Cloudflare) jika origin IP diketahui
    - path yang tidak di-cache CF: /cdn-cgi/trace, /?__cf_chl_jschl_tk__
- Adaptive Logic:
  * Jika response time > 3000ms: turunkan RPS 50%, tambah proxy rotation.
  * Jika error rate > 30%: ganti attack method (GET -> POST -> slowloris).
  * Jika HTTP 403/429: rotate proxy segera, tambah delay jitter 1-3s.
  * Jika HTTP 200 dengan response time normal: naikkan RPS 20%.
  * Real-time metrics: success count, fail count, timeout count, 
    avg response time, current RPS, active proxy count.
- Distributed Node Coordination:
  * Node register ke coordinator (Redis atau shared file) dengan heartbeat.
  * Task distribution: coordinator assign target + method + proxy subset 
    ke tiap node.
  * Node report progress setiap 5 detik: completed, failed, timeout, 
    avg_rtt.
  * Jika node tidak report dalam 15 detik, mark as dead, redistribute task.
