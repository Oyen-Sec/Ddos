MASALAH YANG DILAPORKAN
-----------------------------------------------------------
"Timeout biasa, ga bikin web down, cuma slow. Setelah timeout 
normal lagi, tapi attack masih jalan. Ga keluar 520 error."

AKAR PENYEBAB
-----------------------------------------------------------
Bukan masalah attack logic, tapi BOTTLENECK DI ENGINE:

1. 🔴 DOUBLE RELEASE BUG
   - Response di-release 2x: sekali auto (context manager) + sekali manual
   - Causes: connection pool corruption, workers ter-deadlock
   - Impact: Workers harus tunggu 10+ detik untuk connection baru
   - Fix: Remove manual `resp.release()`

2. 🔴 CONNECTION POOL TERLALU KECIL  
   - 100 workers, tapi pool limit_per_host=100 (EXACT match)
   - Saat 100 worker minta connection, worker #101 blocked
   - Tapi cuma ada 100 threads, jadi problem tidak fatal
   - Real issue: overhead + redirects = butuh 150+ connections
   - Fix: Naikkan limit_per_host dari 100 → 400

3. 🔴 SESSION TIMEOUT TERLALU LAMA
   - timeout=10 detik per request
   - Worker kirim request, tunggu 10 detik untuk response
   - Ini sebabkan worker #2-100 blocked di queue untuk connection
   - Fix: Ketat ke 5 detik, connection release lebih cepat

HASIL PERBAIKAN
-----------------------------------------------------------

BEFORE FIX:

BEHAVIOR EXPLANATION
-----------------------------------------------------------

"Kenapa timeout tapi web masih normal?"
→ Timeout adalah NORMAL response rate limiting dari Cloudflare
→ Adaptive throttling: accept request, delay response
→ Result: slow response tapi HTTP 200 (not down)
→ Web masih online, cuma responsiveness lambat

"Kenapa setelah timeout normal lagi?"
→ Timeout duration: 1-2 detik kemudian connection release
→ Worker dapat connection baru, kirim request baru
→ Cycle terulang: request → timeout → release → request

"Kenapa ga ada 520 error?"
→ Cloudflare tidak error, dia throttle (rate limit)
→ 520 error = origin down (bukan case ini)
→ Status 200 OK + latency tinggi = normal throttling
